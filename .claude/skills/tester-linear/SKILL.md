# QA Check — orchestrate `tester-linear` on a change

## Зачем

Adversarial проверка реализации **поведенческими тестами**, а не чтением диффа. Агент:
- прогоняет релевантные тесты, lint, typecheck, точечное воспроизведение;
- ловит регрессии, missing edge cases, ошибки в null/empty/error path;
- ловит **overfitting** (impl, подогнанная под фикстуры тестов) — то, что ревьюер обычно пропускает;
- при подтверждённом дефекте заводит issue в Linear с шаблоном (Steps to reproduce / Actual / Expected / Evidence / Severity), priority по mapping severity→Linear, assignee всегда unassigned;
- если Linear недоступен — выдаёт готовые блоки для ручной вставки.

Дополняет `dual-review`: тот работает на уровне «как написано», тестировщик — на уровне «что делает». Запускать оба смысла нет — выбирай по задаче.

## Когда НЕ запускать

- В диффе 0 файлов или > 500 файлов → агент сам выйдет с `NEEDS_INFO`, но лучше предупредить пользователя заранее и попросить уточнить scope.
- Только что прогонял тестировщика на этом же SHA (см. дедуп ниже) → спроси «уже было QA на этот же commit, перезапустить?».
- Пользователь просит ревью кода, а не проверку поведения → отправь его на `dual-review` или одиночный `Agent(subagent_type: "reviewer")`.
- Меняли ТОЛЬКО документацию (`*.md`, `docs/**`) → QA не нужно, скажи об этом и предложи `dual-review` если нужна формальная проверка.

## Workflow

### 1. Определи scope проверки

Источники, в порядке приоритета:
- Явная ссылка на PR (`https://github.com/<owner>/<repo>/pull/<num>` или `#<num>`) → возьми `headRefOid` и `baseRefName` для прогона на этом коммите.
- Текущий локальный дифф против `main` (`git diff main...HEAD --stat`).
- Если ничего нет → один вопрос пользователю «что именно проверяем — PR, ветка, конкретный коммит?» и стоп.

### 2. Pre-flight check

```bash
git rev-parse --short HEAD                      # зафиксируй SHA
git diff HEAD~1 --stat                          # есть ли изменения
git diff main...HEAD --stat                     # размер scope
```

Проверь:
- В диффе **что-то есть** (иначе агент сразу выйдет с `NEEDS_INFO`).
- Размер диффа разумный (< 500 файлов). Если больше — предупреди пользователя и попроси сузить.
- Если работаем с PR — `gh pr view <num> --json state,headRefOid,isDraft`. На `MERGED / CLOSED / draft` — спроси подтверждение.

**Дедуп против предыдущего прогона.** Тестировщик создаёт issues с маркером `<!-- qa-marker: sha=<SHA>, finding=<slug> -->`. Перед запуском проверь, не прогоняли ли его уже на этот SHA:

```bash
# Через Linear MCP: list_issues с поиском по `qa-marker: sha=<SHA>`
```

Если найдены свежие issues с этим SHA — спроси «QA на `<SHA>` уже прогонялся, заводил issues `LIN-XXX, LIN-YYY`. Перезапустить или хватит?».

### 3. Резолв Linear-контекста (один раз за сессию)

Linear-team нужен агенту для `create_issue`. Источник, в порядке приоритета:
- env `LINEAR_TEAM` → агент сам резолвит.
- Закэшированный team из предыдущего прогона в этой сессии.
- Иначе — задай пользователю **один** вопрос: «В какую команду Linear заводить дефекты?» и передай ответ агенту.

Не пытайся резолвить самостоятельно через `list_teams` — это ответственность агента, чтобы single source of truth.

### 4. Запусти `tester-linear`

```
Agent({
  subagent_type: "tester-linear",
  description: "QA check on <SHA> / PR #<num>",
  prompt: "Прогонись по своему стандартному алгоритму на текущем диффе.
           SHA: <SHA>. Scope: <PR #<num> URL | local diff against main>.
           Linear team: <team-name-from-user-or-env>.
           Acceptance criteria для проверки (если известны): <...>.
           Если найдёшь подтверждённые дефекты — заводи issues в Linear по своим правилам
           (дедуп через qa-marker, лимит 8, батчинг low → nitpicks).
           Если Linear недоступен — оформи дефекты inline по шаблону для ручной вставки.
           Верни в чат стандартный отчёт: VERDICT, SUMMARY, CHECKS_RUN, FINDINGS,
           LINEAR_DEFECTS_CREATED, LINEAR_DEFECTS_UPDATED, UNCONFIRMED_RISKS, RESIDUAL_RISK."
})
```

### 5. Дождись и собери summary

В отличие от `dual-review` (где отчёт уходит в PR и НЕ дублируется в чат), отчёт тестировщика — это и есть основной артефакт для пользователя. Issues уходят в Linear, отчёт остаётся в чате.

Когда `Agent` вернётся — выдай **сводную мета-информацию + полный отчёт агента**:

```
## QA-прогон на `<SHA>` — <PR title или scope>

### `tester-linear`
- **Verdict:** PASS | PASS_WITH_RISKS | FAIL | NEEDS_INFO
- **Findings:** critical=<n>, high=<n>, medium=<n>, low=<n>
- **Linear:** created=<n>, updated=<n>, batched=<n>, unavailable=<bool>

<полный отчёт агента>
```

Если агент завёл issues — продублируй их список с прямыми ссылками на Linear:
```
**Заведённые дефекты:**
- LIN-XXX (P1, [Bug] …)
- LIN-YYY (P3, [Bug] …)
- LIN-ZZZ (P4, [QA][Nitpicks] <SHA>) — батч nitpicks
```

### 6. Что НЕ делать

- ❌ Не правь код, не «фикси найденные баги» — это работа отдельного шага. Агент тестирует, ты передаёшь отчёт.
- ❌ Не дёргай Linear MCP сам — все `create_issue / create_comment / list_issues` идут через агента, чтобы дедуп и лимиты работали централизованно.
- ❌ Не интерпретируй severity «по своему» — мапинг `critical→1, high→2, medium→3, low→4` фиксирован в спеке агента.
- ❌ Не назначай assignee. Агент создаёт issues unassigned — триаж работа человека.
- ❌ Не запускай тестировщика повторно автоматически на тот же SHA «на всякий случай». Если видишь свежий прогон — спроси.

## Примеры триггеров

- «Запусти тестировщика»
- «Прогон QA на текущем PR»
- «Проверь реализацию на баги»
- «Найди дефекты в этой ветке»
- «@tester-linear»
- «QA-чек перед мержем»

## Чем отличается от `dual-review`

| | `dual-review` | `tester-linear` |
|---|---|---|
| Источник правды | дифф (как написано) | поведение (что делает) |
| Артефакт | review-комменты в PR | issues в Linear + отчёт в чат |
| Ловит | стиль, архитектуру, нарушения CLAUDE.md, регрессии в критпуть демо | падающие тесты, edge cases, overfitting, runtime-ошибки |
| Запуск | после `gh pr create` | после «реализация готова, проверь» |

Запускать оба на один и тот же change смысла нет — выбирай один.

## Зависимости

- Агент: [`tester-linear`](../../agents/tester-linear.md) — должен существовать в `.claude/agents/`.
- CLI: `gh` (для PR-сценария, scope `repo`), `git` (read-only).
- Linear MCP: коннектор должен быть подключён под именем `linear` (или агент упадёт в fallback «Linear unavailable» и выдаст блоки для ручной вставки — это штатная ветка).
