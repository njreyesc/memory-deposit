# ADR-001: Изоляция данных для параллельного пользовательского тестирования (50 человек)

**Status:** Proposed
**Date:** 2026-04-19
**Deciders:** Sim (владелец продукта), архитектор

---

## Context

Прототип "Депозит Памяти" создан для 7-минутного демо одному человеку. Архитектура рассчитана на **единственного оператора**, который последовательно кликает сценарий. Два захардкоженных пользователя (Алексей `11111111-...`, Мария `22222222-...`) живут в `lib/auth/demo-users.ts`, а логин через `/api/mock-login` выдаёт сессию одного из них всем вызывающим.

Новое требование: провести UX-тест с **50 параллельными участниками** на одной ссылке. Каждый должен пройти полный сценарий изолированно — иначе данные смешиваются, видео перезаписываются, а случайный клик "Сбросить демо" уничтожает данные всех 49 остальных.

### Силы в игре

- **Скорость**: тест нужно провести в ближайшие дни, не через месяц.
- **Минимум инвазии**: прототип работает, ломать его ради одного теста опасно.
- **Supabase Free tier**: ~5-10 соединений к БД, 1 ГБ Storage, 50K auth-запросов/мес.
- **Обратная совместимость**: демо-режим для питча должен продолжать работать.
- **Нет тестов/CI**: любая ошибка обнаруживается только руками.

---

## Decision

**Выбран Вариант A: динамические сессии с изолированными парами пользователей.** При входе каждый тестировщик вводит имя, система создаёт ему персональную пару auth-пользователей (breadwinner + recipient) и связывает их через таблицу `test_sessions`. Существующий RLS автоматически изолирует данные по `auth.uid()` / `owner_id`.

---

## Options Considered

### Вариант A: Динамические сессии (test_sessions)

| Dimension | Assessment |
|-----------|------------|
| Сложность | Medium — новая таблица, 2 API-эндпоинта, обновление логина и role-switcher |
| Стоимость | $25/мес (Supabase Pro для 50 соединений) |
| Изоляция | Полная — каждый пользователь имеет уникальные UUID, RLS изолирует автоматически |
| Риск регрессии | Низкий — существующий код не меняется, добавляется параллельный путь |
| Очистка | Простая — DELETE FROM test_sessions CASCADE удаляет всё |

**Как работает:**

```
Тестировщик вводит имя "Иван"
  → POST /api/test-session/create { testName: "Иван" }
  → Создаются 2 auth-пользователя: test-<uuid>-alexey@test.local, test-<uuid>-maria@test.local
  → Создаются 2 записи в users (is_test_user = true, test_session_id = <session_id>)
  → Создаётся test_sessions { breadwinner_user_id, recipient_user_id, session_token }
  → Cookie: test_session_token = <token>
  → Логин как "Иван-Алексей" → все данные привязаны к его UUID
  → Переключение на "Иван-Марию" через тот же session_token
  → Никто другой не видит его данные (RLS: auth.uid() ≠ его UUID)
```

**Pros:**
- Полная изоляция без изменения RLS-политик — они уже работают по `auth.uid()`
- Обратная совместимость — старый демо-путь (Алексей/Мария) не трогается
- Масштабируется — хоть 50, хоть 200 (при достаточных ресурсах Supabase)
- Простая очистка — одна таблица как корень каскадного удаления

**Cons:**
- Нужен Supabase Pro ($25/мес) для 50 параллельных соединений
- Создание 100 auth-пользователей (50×2) занимает ~10-15 секунд при одновременном старте
- 50 видео × 10-20 МБ = 500 МБ-1 ГБ — упирается в лимит Storage

---

### Вариант B: Префиксная изоляция (session_id в данных)

| Dimension | Assessment |
|-----------|------------|
| Сложность | High — изменить ВСЕ запросы, добавить session_id фильтр, переписать RLS |
| Стоимость | $0 (Free tier) |
| Изоляция | Частичная — один auth-пользователь на всех, фильтрация на уровне приложения |
| Риск регрессии | Высокий — меняются все SELECT/INSERT во всех компонентах |
| Очистка | Сложная — DELETE WHERE session_id = X по каждой таблице |

**Как работает:**

Все 50 человек логинятся как один Алексей, но в каждый запрос добавляется `session_id`. Таблица `vault_items` получает колонку `session_id`, RLS фильтрует по ней.

**Pros:**
- Не нужно создавать auth-пользователей — один аккаунт на всех
- Работает на Free tier (1 соединение к БД через Supabase пулер)

**Cons:**
- Переписать ВСЕ RLS-политики (6 таблиц × 4 операции = 24 политики)
- Переписать ВСЕ компоненты — каждый `supabase.from().select()` должен добавлять `.eq('session_id', ...)`
- RPC-функции `recipient_event_status()` и `get_recipient_materials()` — полный рефакторинг
- Один забытый `.eq()` = утечка данных между сессиями
- Role-switcher не работает — auth.uid() один и тот же, нельзя "стать Марией"

---

### Вариант C: 50 отдельных инстансов Vercel

| Dimension | Assessment |
|-----------|------------|
| Сложность | Low (devops) / High (подготовка) |
| Стоимость | $0-25 (Vercel Pro) + 50× Supabase Free проектов = нереально |
| Изоляция | Абсолютная — каждый на своём инстансе |
| Риск регрессии | Нулевой — код не меняется |
| Очистка | Удалить 50 проектов |

**Pros:**
- Ноль изменений в коде
- Абсолютная изоляция

**Cons:**
- Создать и поддерживать 50 Supabase-проектов нереально
- Раздать 50 разных URL — логистический кошмар
- Не масштабируется

## Trade-off Analysis

| Критерий | A: Сессии | B: Префиксы | C: Инстансы |
|----------|-----------|-------------|-------------|
| Изменений в коде | ~8 файлов | ~25 файлов | 0 |
| Риск сломать демо | Низкий | Высокий | Нулевой |
| Время реализации | 1-2 дня | 3-5 дней | 1 день + ops |
| Стоимость/мес | $25 | $0 | Нереально |
| Изоляция | Полная (auth) | Частичная (app) | Абсолютная |
| Очистка | 1 запрос | N запросов | Удалить проекты |

**Вариант A побеждает** по соотношению надёжность/усилия. Ключевое преимущество: RLS уже работает по `auth.uid()`, и динамические пользователи автоматически получают изоляцию без единого изменения в политиках.

---

## Consequences

### Что становится проще
- Провести тест на любое количество участников — масштабируется линейно
- Собрать данные по каждому тестировщику — `test_sessions.test_name` привязано к его данным
- Повторить тест — просто cleanup + новые сессии
- В будущем: если продукт пойдёт дальше демо, `test_sessions` → `workspaces`

### Что становится сложнее
- Нужен Supabase Pro ($25/мес) — Free tier не выдержит 50 соединений
- Нужно следить за Storage (~1 ГБ на 50 видео) — возможно ограничить длину записи до 15 сек
- Создание 100 auth-пользователей при одновременном старте — нужен rate limiting или pre-creation

### Что нужно пересмотреть позже
- После теста: удалить все тестовые данные и auth-пользователей
- Решить, нужна ли эта функциональность для следующих демо или вернуться к single-user mode
- Если продукт идёт в продакшен — `test_sessions` эволюционирует в полноценную multi-tenancy

## Стек (без изменений)

Текущий стек **полностью подходит** для реализации. Никаких новых зависимостей не требуется:

| Компонент | Роль в решении | Изменения |
|-----------|----------------|-----------|
| Next.js 14 App Router | Новый API route `/api/test-session/create` | +2 файла |
| Supabase Auth | `admin.auth.createUser()` для динамических пользователей | Нет изменений |
| Supabase Postgres + RLS | Новая таблица `test_sessions`, существующие RLS работают as-is | +1 миграция |
| Supabase Storage | Пути с префиксом `test_<session_id>/` | Изменение в VideoRecorder |
| Tailwind + shadcn/ui | Новая форма логина | +1 компонент |
| TypeScript strict | Типы для TestSession | +1 файл типов |

**Единственное внешнее изменение:** апгрейд Supabase до Pro ($25/мес).
